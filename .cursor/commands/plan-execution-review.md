## 任务目标
Review本次实现的代码，确保与plan一致，并检查代码质量和架构合理性。

## Review维度
1. **一致性检查**
   - 实现是否与plan中的规划一致
   - 是否遗漏plan中的关键步骤或功能点
   - 实现方式是否符合plan的设计意图

2. **架构与设计**
   - 架构设计是否合理且优雅
   - 是否符合项目的分层架构原则（参考 `docs/ARCHITECTURE.md`）
   - 依赖关系是否正确，是否违反依赖倒置原则

3. **代码质量**
   - 代码逻辑是否清晰、正确
   - 命名是否规范、语义明确
   - 是否遵循项目的编码规范（参考workspace rules）

4. **设计原则**
   - 是否紧贴需求，避免过度设计
   - 是否过度预留扩展性（Engineering Principle：避免推测性扩展）
   - 是否符合"单一职责"、"避免过度设计"等原则

## 输出要求
1. **Review结果报告**：
   - 一致性检查结果
   - 发现的问题列表（如有）
   - 代码质量评估

2. **问题修复**：
   - 对发现的问题进行修复和重构
   - 优先修复架构问题和逻辑错误
   - 代码风格和命名问题次之

## 重要原则
- **不要为了改而改**：如果review后没有明显问题，直接告知结果即可，不要强行修改
- **聚焦实际问题**：只修复真正存在的问题，不要过度优化
