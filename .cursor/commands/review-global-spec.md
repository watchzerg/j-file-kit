## 任务目标
全局检查项目代码是否符合规范，识别问题并按严重程度和影响范围排序，给出解决建议。

## 检查范围

### 1. 循环依赖问题处理

**检查项**：
- **禁止** `from __future__ import annotations`（Python 3.14+ 默认启用）
- **禁止** local import（函数内部的 import）
- 通常不需要 `TYPE_CHECKING`（Python 3.14+ 已解决）
- 自引用应使用 `typing.Self` 更优雅

**输出要求**：
- 列出所有违规项，标注文件路径和行号，说明违反规范的原因和修改建议

### 2. 未使用的代码清理

**检查项**：
- 未使用的模块、类、函数/方法、变量（包括私有变量 `_var`）、类型定义
- **注意**：测试文件中的使用不算有效使用

**输出要求**：
- 按模块分组列出未使用的代码，标注文件路径和行号
- 区分"确定未使用"和"可能未使用"（需要人工确认）

### 3. 命名一致性检查

**检查项**：
- 文件命名：`snake_case`，描述性名称
- 模块命名：符合分层架构（models/、interfaces/、services/、infrastructure/、api/）
- 类命名：`PascalCase`，函数命名：`snake_case`，常量命名：`UPPER_SNAKE_CASE`

**输出要求**：
- 列出所有命名不一致的地方，说明正确的命名，按严重程度排序

### 4. 包的导出结构检查

**检查项**：
- 检查各个包的 `__init__.py` 导出结构是否合理（不考虑历史兼容性，以合理性与结构优雅为优先）

**输出要求**：
- 列出每个包的导出结构问题，说明需要调整的原因和优化建议

### 5. `__init__.py` 和 `__all__` 检查

**检查项**：
- 这是 HTTP 项目，不是 SDK，原则上 `__init__.py` 文件应该**仅保留注释**（模块功能解释说明）
- 一般不需要 `__all__`：能去掉的都去掉

**输出要求**：
- 列出所有不符合规范的 `__init__.py` 文件和 `__all__` 使用位置
- 说明应该如何处理（移除导入、移除 `__all__`、仅保留注释）

## 输出格式

### 问题分类和排序

按以下维度对问题进行分类和排序：

1. **严重程度**：
   - **严重**：违反核心规范，可能导致运行时问题或架构问题
   - **中等**：违反规范，影响代码质量，但不影响功能
   - **轻微**：风格问题，影响可读性

2. **影响范围**：
   - **全局**：影响整个项目或核心架构
   - **模块**：影响单个模块或包
   - **局部**：影响单个文件或函数

3. **修复优先级**：
   - **P0**：必须立即修复（严重 + 全局）
   - **P1**：应该尽快修复（严重 + 模块，或中等 + 全局）
   - **P2**：建议修复（中等 + 模块，或轻微 + 全局）
   - **P3**：可选修复（轻微 + 局部）

### 输出结构

按优先级（P0/P1/P2/P3）列出问题，每个问题包含：文件路径、问题描述、影响范围、修复建议。

**重要原则**：如果检查后没有发现明显问题，应直接指出"无明显需要改动的地方"，不需要强行找出问题或过度优化。

## 优化建议和方案对比

对于发现的问题，需要提供优化建议和方案对比：

1. **结合业界最佳实践**：参考主流 Python 项目做法，遵循 PEP 规范，考虑项目架构设计原则（参考 `docs/ARCHITECTURE.md`）

2. **提供多个方案（如适用）**：
   - 如果存在多种可行方案，列出 1-3 个主流、优雅的方案
   - 如果存在一种明显更优的方案，则仅列出这一种方案即可
   - 每个方案应包含：核心思路、优缺点、改造量（小/中/大）、风险（低/中/高）、收益、适用场景

3. **推荐方案**：如果有多种方案，应推荐最适合当前场景的方案，说明推荐理由

**客观评估原则**：
- 不要讨好用户思路，如果用户提出的方案不合理，应明确指出
- 提供理由：指出问题时，必须说明理由和解释
- 不要为了改而改：如果检查后没有明显问题，直接告知结果即可，不要强行修改

## 检查工具建议

- `ruff check --select F401`：检查未使用的导入
- `ruff check --select F841`：检查未使用的变量
- `mypy --strict`：类型检查
- `vulture`：查找未使用的代码
- 使用 `grep` 搜索特定模式，使用 `ast` 模块解析 Python 代码

## 注意事项

- 某些"未使用"的代码可能是为了 API 兼容性或未来扩展，需要结合业务逻辑判断，不确定的情况标注为"需要人工确认"
- 测试文件中的使用不算有效使用
- 优先修复严重问题，可以分批次修复，修复后需要运行测试确保功能正常
- 所有修复建议应符合项目架构设计（参考 `docs/ARCHITECTURE.md`）和编码规范
