## 任务目标
分析指定目录或代码文件，理解设计意图，编写对应的单元测试用例。

## 核心原则
- **基于设计意图**：测试反映真实设计意图，而非实现细节
- **适度覆盖**：完整覆盖关键路径和重要场景，避免过度测试
- **发现问题先暂停**：遇到问题先与用户确认，不直接修复

## 测试要求

### 代码理解
- 深入理解设计意图和业务逻辑，参考注释和文档
- 识别核心功能和边界条件

### 测试覆盖
- 包含正常流程、边界条件、异常场景
- 聚焦关键路径，避免过度测试

### 测试组织
- 遵循项目测试目录结构规范
- 测试文件命名清晰，与源文件对应
- 使用 `pytest` 框架，添加清晰的 markers（如 `@pytest.mark.unit`）
- 使用工厂/构建器模式生成测试数据
- 测试应独立、可重复、快速执行

## 分层测试原则

### Model 层
- **简单容器无需测试**：纯数据容器（如 Pydantic 模型仅做字段定义和验证）可不测试
- **关注业务逻辑**：仅对包含业务逻辑的 model 方法进行单元测试

### Interface 层
- **通常无需测试**：interface 层作为抽象定义，通常不需要单元测试

### API 层
- **不放置业务逻辑**：API 层应保持薄层，仅做请求/响应转换和路由
- **不进行单测**：API 层的正确性通过集成测试验证

### Infrastructure 层 - 数据库
- **SQLite**：使用 `:memory:` 创建临时数据库，每个测试用例使用独立的数据库实例
- **PostgreSQL 等外部数据库**：
  - 将 `_row_to_model` 等数据转换方法拆分为独立函数，便于单元测试
  - 单元测试仅覆盖数据转换逻辑，SQL 正确性归属集成测试
  - 避免在单元测试中连接真实外部数据库

### Infrastructure 层 - 文件系统
- 使用 `tmp_path` 在隔离的临时目录中进行测试
- 每个测试用例使用独立的临时目录，确保测试隔离

## Fixture 最佳实践
- 按功能模块组织 fixtures（如 `tests/fixtures/models.py`、`tests/fixtures/mocks.py`）
- 使用 `conftest.py` 放置共享 fixtures，避免全局命名空间污染
- 使用 `pytest.fixture` 装饰器，明确作用域（`function`、`class`、`module`、`session`）
- 数据库 fixtures 使用 `function` 作用域，确保测试隔离
- 复杂 fixtures 使用工厂模式（`@pytest.fixture` 返回工厂函数）

## 问题处理
如发现以下问题，先暂停并提示用户：
- 设计不清晰，无法理解代码意图
- 注释模糊或缺失，影响测试编写
- 明显的代码 bug 或逻辑错误

**处理方式**：说明问题和影响，给出修改建议（不直接修复），等待用户确认。
