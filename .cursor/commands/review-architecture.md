## 任务目标
对项目整体架构进行深度审查，分析架构设计（模块定位、依赖关系、分层等）是否符合业界最佳实践，识别潜在问题并给出优化建议。

## 审查范围
1. **架构分层设计**
   - 各层的职责定位是否清晰
   - 层间依赖关系是否合理
   - 是否存在跨层调用或循环依赖
   - 分层是否符合Clean Architecture等主流架构模式

2. **模块设计**
   - 模块职责是否单一
   - 模块间耦合度是否合理
   - 模块边界是否清晰
   - 是否存在过度设计或设计不足

3. **依赖管理**
   - 依赖方向是否符合依赖倒置原则
   - 是否存在不必要的依赖
   - 接口抽象是否合理
   - 依赖注入机制是否完善

4. **扩展性设计**
   - 是否支持未来功能扩展
   - 扩展点设计是否合理
   - 是否存在过度设计（违反YAGNI原则）

5. **数据流设计**
   - 数据流向是否清晰
   - 是否存在数据冗余或不一致
   - 持久化设计是否合理

## 审查方法
1. **阅读项目文档**
   - 仔细阅读 `docs/ARCHITECTURE.md` 了解架构设计
   - 阅读 `README.md` 了解项目目标和设计目的
   - 参考项目编码规范和设计原则

2. **代码结构分析**
   - 分析目录结构和模块组织
   - 检查各层的实际实现
   - 验证架构文档与实际代码的一致性

3. **依赖关系分析**
   - 分析模块间的导入关系
   - 检查是否存在循环依赖
   - 验证依赖方向是否符合架构设计

4. **业界最佳实践对比**
   - 对比 Clean Architecture、Hexagonal Architecture等主流架构模式
   - 参考 Python 项目的最佳实践
   - 考虑项目规模和复杂度是否匹配当前架构

## 输出要求
1. **架构分析报告**
   - 项目设计目的和架构目标总结
   - 当前架构设计概述（分层、模块、依赖关系）
   - 架构设计优点和亮点

2. **问题识别**
   - 发现的问题列表（按严重程度和影响范围排序）
   - 每个问题的详细说明（问题描述、影响范围、潜在风险）

3. **优化建议**
   - 针对每个问题给出优化方案
   - 每个优化方案需包含：
     - **方案描述**：具体的优化措施
     - **优缺点**：方案的优点和缺点
     - **改造量**：需要修改的文件数量、代码行数、影响范围
     - **风险**：实施风险（如：破坏性变更、测试成本、回滚难度）
     - **收益**：预期收益（如：可维护性提升、性能改善、扩展性增强）
   - 多个方案时需进行对比分析

4. **结论**
   - 如果架构没有明显问题，直接指出：**无明显需要改动的地方**
   - 如果有问题，给出优先级建议和实施建议

## 重要原则
- **客观分析**：基于实际代码和架构文档进行分析，避免主观臆断
- **适度优化**：遵循项目原则"避免过度设计"，只针对实际问题提出优化
- **实用主义**：优化建议需考虑项目实际情况（规模、团队、时间成本）
- **对比分析**：多个方案时需全面对比，帮助决策
- **风险意识**：充分评估改造风险和收益，避免为优化而优化
